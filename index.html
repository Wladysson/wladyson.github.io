  • Na próxima atualização, o Suprema ADMIN incluirá funções de busca ativa de novos pacientes, com integração a plataformas externas, redes de saúde e canais digitais. Essa nova funcionalidade tem como objetivo aumentar significativamente o fluxo de pacientes e ampliar o alcance da clínica.

Plataforma Financeira

A solução funciona em arquitetura Java com processamento assíncrono e comunicação por Kafka, garantindo consistência com o padrão SAGA e registro imutável de eventos para fins regulatórios. A infraestrutura é criada com Terraform, usando EKS, RDS, VPC, Vault, CI/CD com GitHub Actions e GitOps com ArgoCD, além de observabilidade com Prometheus, Grafana e Alertmanager. Na fase inicial são implementadas as bases do ambiente e o módulo de contas, incluindo CRUD completo e integração transacional com PostgreSQL. Em seguida é construído o módulo de transações, responsável por PIX, TED e DOC, usando eventos no Kafka e garantindo exatamente uma vez no processamento. Depois são desenvolvidos os módulos de cartões e conciliação, com rotinas de processamento diário e fluxos em tempo real. Na fase final entram o sistema de auditoria imutável, otimizações de desempenho e testes de resiliência com falhas controladas no Kubernetes, junto com a finalização dos dashboards de SLO e SLI para acompanhamento do comportamento dos microsserviços. Os processos de fechamento contábil executam cálculos e agregações noturnas, enquanto a trilha de auditoria salva todos os eventos de forma permanente para cumprir exigências regulatórias.

Sistema de Gateway de Pagamentos Assincronos

  • Projeto feito para cenários de alta concorrência e baixa latência, inspirado em sistemas do tipo PIX. A plataforma recebe requisições de pagamento e executa o processamento de forma totalmente assíncrona, garantindo atomicidade lógica e consistência eventual entre contas. A arquitetura é orientada a eventos e composta por dois microsserviços principais: um Gateway Handler, responsável pela ingestão, validação e publicação dos eventos de pagamento, e um Transaction Processor, responsável pela execução dos débitos e créditos. A coordenação entre os serviços é realizada por meio do padrão SAGA por coreografia, permitindo compensações automáticas em caso de falhas parciais. A comunicação entre os componentes ocorre via Apache Kafka, assegurando entrega confiável, ordenação e tolerância a falhas durante o processamento. O sistema é executado sobre Kubernetes, garantindo orquestração, escalabilidade e alta disponibilidade. O Gateway Handler foi desenvolvido em Java com Quarkus Reativo, utilizando Mutiny para programação não bloqueante, permitindo milhares de conexões simultâneas com baixa latência. Métricas críticas de desempenho, incluindo latência P99, são monitoradas continuamente por meio do Prometheus. A persistência utiliza PostgreSQL como banco transacional, provisionado via Terraform, com configuração explícita de níveis de isolamento e controle de locks para preservação da integridade transacional em cenários concorrentes.

Motor de conciliação de Contas (Batches e Streams)

  • Uma Aplicação projetada para comparar grandes volumes de transações internas com extratos bancários, combinando processamento em lote e em tempo real para garantir precisão de saldos e identificação confiável de divergências. O sistema realiza conciliação batch por meio de jobs complexos implementados em Java com Spring Batch, otimizados para leitura, processamento e escrita de arquivos de grande porte, incluindo extratos bancários com até 10 GB. A execução recorrente desses processos é orquestrada via Kubernetes CronJobs, assegurando previsibilidade e isolamento de carga. Para conciliações contínuas, o motor utiliza Kafka Streams para processar eventos financeiros em tempo real, permitindo a reconciliação imediata de transações, estornos e correções à medida que os eventos são publicados, eliminando a dependência exclusiva do batch diário. Os arquivos brutos são armazenados no AWS S3, garantindo durabilidade e rastreabilidade, enquanto os resultados consolidados de conciliação são persistidos no AWS Redshift, possibilitando análises históricas e consultas analíticas em larga escala. Toda a infraestrutura foi provisionada como código com Terraform, assegurando ambientes reprodutíveis e consistentes. A observabilidade do sistema foi estruturada com Datadog, monitorando a saúde dos workers, o throughput de processamento e o lag das filas Kafka, permitindo detecção proativa de gargalos e falhas operacionais.

Suprema Admin - Sistema de Automação Administrativa de Saúde

  • Sistema de automação e processamento de dados voltada à gestão administrativa de clínicas e instituições de saúde, projetada para tratar grandes volumes de informações não estruturadas e integrá-las a sistemas oficiais de forma automática, precisa e auditável. O sistema realiza a ingestão e interpretação de dados a partir de fontes totalmente desestruturadas, incluindo PDFs, planilhas, fichas de atendimento, laudos médicos, formulários manuais e relatórios diversos. A partir desses insumos, a plataforma extrai, organiza e estrutura as informações, gerando planilhas padronizadas e relatórios detalhados com horários de atendimento, observações clínicas e dados prontos para análise e validação, eliminando a necessidade de preenchimento manual. Além da organização dos dados, a plataforma executa automação completa de digitação em sistemas oficiais, reconhecendo campos, validando informações e preenchendo formulários externos com alta precisão. Em cenários de inconsistência ou ausência de dados, o sistema emite alertas, bloqueia o envio de informações incorretas e destaca automaticamente os campos problemáticos, garantindo conformidade e reduzindo erros operacionais. A solução automatiza também a identificação de planilhas e relatórios exigidos por diferentes órgãos reguladores, realizando a extração das informações relevantes e o envio conforme prazos e formatos específicos, eliminando processos manuais e o envio por e-mail. Esse fluxo garantiu padronização, rastreabilidade e conformidade regulatória contínua. Como resultado operacional, o sistema executou 12.526 procedimentos digitados automaticamente em apenas 5 dias, um volume que anteriormente demandava entre 45 e 50 dias úteis de trabalho manual, realizado por uma equipe de aproximadamente 6 pessoas. A automação reduziu drasticamente o tempo de processamento e liberou a equipe administrativa para atividades estratégicas.

API de Monitoramento de Limites e Fraude

  • Este projeto foca na validação e tomada de decisão em tempo real para evitar fraudes e estouro de limites durante transações. O objetivo foi criar uma API Java de alto throughput que, a cada transação recebida, consulta o histórico do cliente e aplica regras de negócio complexas — como limites diários e frequência de uso — em poucos milissegundos. Para garantir decisões rápidas, utilizei Redis (via Caffeine como cache local + Redis remoto) para armazenar limites e dados recentes do cliente em memória, mantendo tempos de resposta abaixo de 50ms. O cluster Redis foi provisionado com Terraform usando AWS ElastiCache. As regras de negócio foram implementadas com Drools, permitindo atualizar regras de fraude sem recompilar o código principal. O pipeline de CI/CD no GitHub Actions valida e implanta novas regras em um ambiente Blue/Green no Kubernetes, garantindo atualizações seguras. O monitoramento foi feito com Prometheus e Grafana, criando dashboards que acompanham as taxas de “Rejeitadas por Limite” e “Rejeitadas por Fraude”, alinhadas às métricas de negócio definidas segundo práticas de SRE.

Sistema de Gerenciamento de Limites de Crédito em Tempo Real

  • Em fintechs, o cálculo e a validação de limites de crédito precisam acontecer em milissegundos para evitar overdraft e fraude. Implementei o serviço usando Quarkus com endpoints reativos via Mutiny, garantindo inicialização rápida, baixo consumo de memória e resposta imediata. Isso melhora a eficiência dos pods no Kubernetes (EKS) e reduz custos. No armazenamento de limites usei Caffeine como cache local e AWS ElastiCache (Redis) como cache distribuído, aplicando uma estratégia cache-aside avançada. O Redis foi provisionado via Terraform, e o serviço Java inclui fallback e failover caso o cache fique indisponível. Na integridades implementei o padrão de Two-Phase Commit para manter consistência estrita entre o serviço de limite e o serviço de débito/crédito, atendendo exigências regulatórias de integridade de dados e no testes de performance executei testes de carga em larga escala com Gatling, integrado ao pipeline de CI/CD no GitHub Actions, garantindo que problemas de performance sejam detectados antes da implantação em produção.
